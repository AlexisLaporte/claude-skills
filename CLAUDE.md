# claude-skills

Repository of generic, reusable skills for Claude Code. Each skill is a directory with a `SKILL.md`; users install via symlink into `~/.claude/skills/`.

See `README.md` for the user-facing skill list and other skill sources.

## Architecture

```
claude-skills/
├── README.md           # Public skill index, installation note
├── skills/             # One directory per skill
│   └── <slug>/
│       ├── SKILL.md    # Required — frontmatter + body
│       └── scripts/    # Optional — helpers invoked by the skill
├── scripts/
│   └── build-api.sh    # Aggregates all SKILL.md frontmatter into JSON
└── dist/api/           # Generated by build-api.sh
    ├── skills.json     # Flat array of all skills
    └── skills/<slug>.json
```

## SKILL.md frontmatter

```yaml
---
name: <slug>                   # Falls back to dir name if absent
description: <one line>        # Used by the model to decide relevance
disable-model-invocation: false
argument-hint: "[arg1|arg2]"   # Optional, displayed in autocomplete
---
```

The `name` field is what users type as `/<name>`. It can differ from the directory name — e.g. `skills/session/` exposes itself as `session-lookup`.

## Commands

```bash
./scripts/build-api.sh   # Regenerate dist/api/ JSON from all SKILL.md
```

No tests, no lint — skills are markdown.

## Conventions

- **Slug = directory name** when possible. Use a different `name:` only when there's a meaningful reason (e.g. the file system slug is generic but the skill needs a specific public name).
- **Description** must be specific enough for the model to decide whether to invoke. Avoid vague descriptions like "helper for stuff".
- **No legacy fallbacks** in skill bodies — if a skill is renamed, update references rather than aliasing.
- **Keep SKILL.md under ~200 lines** unless the skill genuinely needs more. Long instructions = the model picks up less of it.

## Installation pattern

User-local install:

```bash
ln -sf /data/projects/claude-skills/skills/<slug> ~/.claude/skills/<slug>
```

Other repos contribute skills the same way (`/data/oto/skills/`, `/data/howto/skills/`, etc.) — see README's "Other skill sources" table.

## Adding a new skill

1. Create `skills/<slug>/SKILL.md` with frontmatter + body
2. Add the row to README's `## Skills` table
3. Symlink into `~/.claude/skills/<slug>` to test locally
4. Run `./scripts/build-api.sh` if the JSON API is consumed externally

## Renaming a skill

1. `git mv skills/<old> skills/<new>`
2. Update the `name:` field in `SKILL.md`
3. Update README
4. Replace the symlink: `rm ~/.claude/skills/<old> && ln -sf .../skills/<new> ~/.claude/skills/<new>`
